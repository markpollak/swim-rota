## A. What the app is (context for the findings)

A mobile-first PWA for pool staff — swim teachers and lifeguards — to self-allocate to
half-hour poolside shifts, with admin approval. FastAPI + SQLite backend (~2,600 LOC),
vanilla-JS single-file frontend (~3,100 LOC), no build step, three runtime dependencies
(`fastapi`, `uvicorn`, `tzdata`). Deployed with Docker + Caddy behind Cloudflare on a
$6/month droplet.

**Why the stack holds up:** one pool, tens of staff, single-digit concurrent users. SQLite
in WAL mode with a single uvicorn worker is comfortably inside its envelope, and the
absence of an ORM, a bundler, and a JS framework means the entire system is readable in an
afternoon — which is exactly what makes five audit passes in a week possible.

## B. Concurrency — confirmed still sound

The app's core promise is *who holds which shift*. Reviews 1–2 found and fixed two real
races in the booking path (lost-update slot claims; double-booking bypass). Review 5
re-verified the protections are intact and complete:

- Every check-then-write path (`request`, `approve`, `assign`, `bulk-assign`, `delete`,
  `restore`, schedule `reconcile`) runs inside `BEGIN IMMEDIATE` (`db.get_db_immediate()`),
  taking SQLite's write lock *before* the read.
- Every state transition is a **conditional UPDATE** (`WHERE id=? AND status='open' AND
  deleted_at IS NULL` etc.) with a `rowcount` check → a lost race returns **409**, never a
  false success.
- The races are regression-tested with genuinely parallel HTTP requests against a live
  uvicorn (barrier-released threads), not a serialised test client.

Two stragglers had escaped the pattern — both fixed this pass (R5-2 below).

## C. Review 5 — fresh findings (all fixed)

### 🟠 R5-1 — Stored XSS via the rota-builder tooltip (frontend)

The rota grid escapes every visible label through `esc()` — 134 call sites, previously
audited clean. But one **`title` attribute** interpolated the assigned person's name, the
role name, and the class name raw:

```js
title="${taken ? (name || s.status) + " · " : ""}${role ? role.name + " · " : ""}${s.level_name || "Pool duty"}"
```

`PATCH /api/me` lets any staff member set their own `full_name`. A name containing `"`
breaks out of the attribute and can inject arbitrary attributes (e.g. an event handler)
into the **admin's** rota-builder view — a stored XSS from any staff account targeting
admins. The CSP at the Caddy layer narrows what a payload can do, but the hole is real.

**Fixed:** all three interpolations wrapped in `esc()` (service-worker cache bumped to
v62). This was the only unescaped sink found in a fresh scan of all 74 `innerHTML`
assignments and every templated attribute.

### 🟡 R5-2 — `reject` and `release` were the last unguarded check-then-writes

`reject_slot` and `release_slot` still used a plain transaction and an **unconditional**
`UPDATE … WHERE id=?`. Two admins working the same approvals queue could interleave
approve/reject so the reject lands *after* the approve and silently un-approves the shift
(the approver sees success; the worker's shift evaporates). Similarly a release racing a
re-assign could wipe the new holder.

**Fixed:** both now use `get_db_immediate()` + conditional UPDATEs guarding `status` and
`assigned_user_id`, returning 409 ("That request changed before it could be declined.")
when the row moved underneath them — the same pattern as the rest of the booking path.

### 🟡 R5-3 — Any account could list any channel's members

`GET /api/channels/{cid}/members` had no membership check (unlike `GET …/messages`, which
has one). Any authenticated staff account could enumerate channel IDs and read the full
membership of every channel — including other people's admin-DM channels. Low-impact
information disclosure inside a closed staff group, but inconsistent with the messages
rule. **Fixed:** members or admins only (403 otherwise) — now tested.

### 🟡 R5-4 — Archived channels accepted new messages via the raw API

Deleting a channel soft-deletes it (`active=0`, kept in the admin archive), but
`POST /api/channels/{cid}/messages` never checked `active`. A member with a stale tab (or
curl) could keep posting into an archived channel — invisible to the UI, persisted in the
archive's history. **Fixed:** posting to an inactive channel returns 404 — now tested.

### 🟡 R5-5 — Deactivated *roles* kept generating shifts

Review 3 (S6) made the generators skip **inactive levels** — but not inactive **roles**.
Retiring a role in Manage → Roles left every class/template referencing it silently
generating unfillable shifts for that role, 26 weeks ahead, forever. **Fixed:** both
generators (`generate_slots`, `generate_from_schedules`) now join `roles` and require
`active=1` — covered by a new test (deactivate → generate → 0 shifts; reactivate →
generate → shifts return).

### 🟢 R5-6 — Horizon roll-forward used the container's clock, not the club's

Review 2 (D1) moved all "today" logic to the configured timezone — except
`extend_to_horizon` in `scheduling.py`, which still used `date.today()` (UTC in the
container). Worst case the bookable window drifts by a day around midnight. **Fixed:**
the callers now pass `today_local()` in.

### 🟢 R5-7 — `training_expiry` accepted any string

Training status is computed by *string-comparing* ISO dates, so a value like
`"31/12/2026"` (saved without complaint via `PATCH /api/me` or the admin user editor)
would silently misclassify a lifeguard's training status. **Fixed:** all three write
paths validate `YYYY-MM-DD` → 400 — now tested.

### 🟢 R5-8 — `/api/generate` weeks parameter unvalidated and unbounded

Garbage → 500; `{"weeks": 5000}` → ~96 years of slot generation in one request (an
admin-only self-own, but a single mistyped digit away). **Fixed:** 400 on non-numeric,
clamped to 1–52.

### 🟢 R5-9 — CSV formula injection in the outstanding-shifts export

Class, role and staff names are admin/staff-entered and went into the CSV verbatim; a
value starting `=` / `+` / `-` / `@` executes as a formula when the export is opened in
Excel. **Fixed:** such cells are prefixed with `'` (`_csv_safe`).

### 🟢 R5-10 — `PATCH /api/settings` accepted arbitrary keys

Any admin could insert unknown settings rows (at best clutter, at worst a foot-gun as the
settings surface grows). **Fixed:** whitelist (`timezone`, `demo_logins`); unknown keys →
400.

### 📝 Doc nit

`CLAUDE.md` said the service-worker cache version was v50; the file was at v61. The
single-source-of-truth refactor (Review 2, D4) means nothing breaks — but the doc now says
v62 to match reality.

## D. Security posture (cumulative)

- **Passwords:** PBKDF2-HMAC-SHA256, 120k iterations, per-user salt (stdlib only).
- **Tokens:** HMAC-signed bearer tokens, 30-day TTL; deactivating a user invalidates
  access immediately (`active=1` checked on every request).
- **Login:** rate-limited per real client IP (`CF-Connecting-IP` aware, so it can't be
  spoofed past Cloudflare nor collapsed onto the edge IP); seeded admins still on the
  default password are forced to change it; demo credentials hidden unless explicitly
  enabled.
- **SQLi:** every query parameterised; dynamic fragments built only from hard-coded
  column whitelists.
- **XSS:** `esc()` discipline across the frontend; the one unescaped attribute found this
  pass is fixed. CSP, HSTS, nosniff and frame-deny headers set at the proxy.
- **Transport/origin:** Cloudflare Full (strict) with an origin cert; origin-firewall
  runbook documented (P2).
- **Known accepted risks:** SSE auth token travels as a query parameter (EventSource
  can't set headers) — mitigated by HTTPS end-to-end and short log retention; no
  per-device token revocation (rotate `SWIM_SECRET` to log everyone out); no enforced
  FK constraints (soft-delete-dominant schema, documented as intentional).

## E. Performance & scalability (unchanged, by design)

Real-time messaging fan-out and the login-rate-limit map are in-process → the app is
correct **only with one uvicorn worker** (the Dockerfile pins this). One pool fits in a
fraction of that budget; multi-pool would mean Postgres + a shared broker — a different
product, deliberately deferred. The 26-week slot horizon rolls forward daily in-process.

## H. Remaining roadmap (none blocking)

1. **Role-removal reconciliation** — removing a *role* from a person doesn't re-validate
   shifts they already hold for it (deactivating the whole person does). Same treatment
   as M1 would close it.
2. **Real FK constraints** (C3) — deferred; retrofitting needs table rebuilds on live
   data for little benefit given soft-deletes.
3. **Multi-pool architecture** (E1) — Postgres + Redis pub/sub, only if it ever needs to
   be a product.
4. Legacy `templates` endpoints predate the schedule editor and have lighter validation —
   candidates for removal now schedules are the single source of truth.

## I. Bottom line

Five passes in, the audit is finding tooltip-escaping and two-admins-racing edge cases
rather than "two people both own the shift" — which is what *done* looks like for an app
this size. The codebase remains small enough to re-audit end-to-end in a sitting, the
test suite now guards the bugs that matter most, and every fix in this pass shipped with
a test or a verified repro. Staff can trust it.
